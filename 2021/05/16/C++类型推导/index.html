C++类型推导

发表于 2021-05-16 更新于 2021-05-17 分类于 C++ ， modern C++ ， 类型推导
现代C++的一个重要特点是能够实现类型自动推导，无论是使用 auto 显式指出类型推导需求，还是模板当中的类型推导，它们有些隐秘的规则，使用时要仔细考虑。

类型推导

首先，考虑能够传到一个模板函数的参数的参数类型，显然的是，类型共有三种，分别是：

  • ParamType 具有指针或引用类型，但不是万能引用
  • ParamType 是一个万能引用
  • ParamType 既非指针也非引用

接下来，讨论这三种情况下， 模板的表现。

ParamType 具有指针或引用类型，但不是万能引用

  1. 首先，若 expr 具有引用性质，则先将引用部分忽略
  2. 将 expr 的类型与 ParamType 执行模式匹配，来决定 T 的类型。

对比这两个可以看出，当模板带有 const 时，T 的类型被推导为不带 const 的类型，这是因为一个作为传入的参数 x, xx, rx，其已经具有 const 性质，所以 T 就不需要携带这个性质。

这上面的结果都比较显然，那我们就试试一些新的东西，那么在第二种情况下呢？

ParamType 是一个万能引用

如果是一个万能引用，那么就需要考虑 expr 是左值还是右值。

  • 如果 expr 是一个左值，那么 T 和 paramType 都被推导为左值引用（这是模板推导中，T被推导为引用的唯一情形
  • 如果 expr 是一个右值，那么同情况1

ParamType既非指针也非引用

当 ParamType 不是指针和引用时，一个显然的结论是，param 会是一个实参的复制，即按值传递。对于一些基础类型，按值传递是没有问题的，但是如果传入的是一个指针，那么按值传递就等于一个浅拷贝，来实际看看会发生什么吧。

来思考一下这是为什么，如果是按值传递的话，那么函数的内容就是传入的参数的一个拷贝，那么const或volatile性质也变得不重要了，显然，引用特性也要被忽略。

auto 类型推导

那么，谈到auto类型推导的时候，我们想要的是什么行为呢？首先，我们想要使用auto类型推导的变量、函数返回值等等符合我们的预期。可是 auto 类型推导是怎么做的呢？

实际上，auto 类型推导所做的事情和模板类型推导相似。

同样的，auto 的类型推导也是以三种情况作为分界线：

  • 类型饰词是指针或引用，但不是万能引用
  • 类型饰词是万能引用
  • 类型饰词既非指针也非引用

但是 auto 类型推导却有一个不得不说的问题，即它推导的以大括号为初始值的变量，类型是 std::initializer_list，说起来有些玄学，特别是以大括号作为类的初始值时，编译器也会尽量调用参数是 std::initializer_list 的构造函数，这个问题会放在后面讨论。

可以看出的是，如果使用大括号作为一个变量的初始值，那么大括号内的数据要是同类型的，即要同是 int 类型，因为在 std::initializer_list 模板下，如果是不同的类型，那么模板的参数的类型就无法确定。

还有一些情况会用到 auto，例如 lambda式或函数返回值，在这两种情况下使用 auto 会使用模板类型推导的规则。

尽量使用 auto 声明，而非显式声明

auto 的类型自动推导功能给我们带来了极大的便利，可是我们对于程序的掌控欲总会让我们显式的声明变量类型，而不是把这个任务交给编译器处理。可是有时候我们未必比编译器做的好，特别是一个程序成为我们的心智负担的时候。

那么就讨论一下，为什么要使用 auto

一个重要的原因就是现代的C++代码已经比较庞大，特别是STL和一些库，有可能它的内部隐藏了一部分东西，如果采用显式声明的方法，有可能会犯错：

看上去人畜无害的代码，可是真的是这样吗，事实上，std::unordered_map 的键值是 const，但是 p 的第一部分为 std::string，我们如果要实现这样的东西，编译器就不得不找到办法从一个 const 变向 non-const。

还有一些函数的返回值问题，由于 C/C++ 的历史悠久，它经历了32位机器和64位机器，我们习惯于64位机器编程，可是一些内置类型在它们这两种机器的表现可能不同，例如，unsigned 和一些别的变量类型，如果移植之后可能会出现一些问题，所以为了不出现这些问题，我们也可以尽量使用 auto。

但是何时不适合使用 auto 呢？一个前面已经提出的问题是，例如使用大括号初始化变量，那么这个变量就是 std::initializer_list 的，这时候，如果这个东西不是我们想要的，我们就尽量使用小括号初始化，或者使用显式声明。

还有一些比较特殊的地方，例如“Invisible” proxy types（隐形的代理类型）会导致 auto 推导的类型不是我们所期望的。一个比较著名的案例是 std::vector\，按理来说，bool 变量应该是一个 bit，但是 std::vector::operator[] 会返回元素的引用，可是 C/C++ 不允许返回 bit 的引用，那他们是怎么做的呢？通过 [] 返回的是个 std::vector\::reference 类型的对象

所以，如果使用 auto b = m[2]，显然， b 的类型是 std::vector\::reference，这不是我们想要的。

std::vector\::reference 是个代理类的实例，所谓代理类，就是指为了模拟或增广其它类型的类。其实，标准库的智能指针也是代理类。

decltype

说完了模板和auto的类型推导，那么还有一个类型推导的关键字，decltype。它的作用是什么呢？在大部分的情况下，它直接返回变量的类型。

如果要考虑到一个特殊情况，我们就试试用auto作为函数的返回值吧，在这个时候，auto使用的是模板类型推导的规则，如上面说的，模板在类型推导的时候会隐去引用，如果我们要返回一个引用的话，会不会出问题呢？
