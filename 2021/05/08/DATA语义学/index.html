C++类的DATA语意学

发表于 2021-05-08 更新于 2021-05-11 分类于 C++ ， class ， 语意学
C++类的DATA按照什么内存模型进行存储，如果有继承的时候，DATA模型又会是什么样的，C++类的DATA能够像普通的内存一样通过指针访问吗？如果可以，效率如何？

Data Member Layout (布局)

如果是一个无继承、无 virtual function 的类，那么它的 Data Member 的分布和C的 struct 相同，虽然可能存在不同的 section，但是一般来说，编译器会将不同的访问权限的内容进行合并，然后按照一定的顺序进行排列。但是如果加上 virtual function 和继承之后，事情将变得不同，特别是加上 virtual base class。

单一继承下的 class

如果是单一继承下的 class，如果没有多态，即没有 virtual function，那么内存模型是正常的堆砌，但是有一点需要注意，那就是内存的进行对齐所填补的内容

以一个实例来解释：

这种模型下，如果不考虑字节的对齐，如果 int 占用 4 字节，那么应该占用 8 字节（考虑一个字节的对齐）

但是实际上，它占用了 16 字节。

这是为什么呢？主要还是因为指针，因为我们不知道一个 Concrete1 * A 会指向什么类型的 class。如果一个 base class 的指针指向了 derived class，如果不采用上面的模型，那么就会发什么奇奇怪怪的错误。

单 class 加上 virtual function

当加上 virtual function 会发生什么呢？众所周知，如果加上 virtual function，那么一般的编译器会产生两个东西，一个是 virtual table(vtbl)，一个是 vptr，vptr 是需要在 Class 模型里面的，因为如果要访问 virtual table，通过类里面的 vptr 就能访问到。

但是 vptr 要放在哪里？事实上，放在哪里都可以，但是放在不同的地方会带来不同的效率影响。一般来说都会放在 class 的头部或者尾部，分析一下这两个部位分别有什么优点和缺点：

尾部：

放在尾部的话，那么 class 的内存模型除了 vptr 外和 C struct 是相同的，所以说，如果这样放，是能够适应以前的 C 代码。但是缺点呢，在多重继承下，不仅从 class object 起始点开始量起的 offset 必须在执行期备妥，甚至与 class vptr 之间的 offset 也必须备妥。（这里需要引入多重继承下，尾端 vptr 的 class 内存模型，但是这种模型现在使用的不多，这里仅仅作为一个介绍了。

头部：

在多重继承下，通过指向 class members 的指针调用 virtual function，但是缺点就是丧失了 C 的兼容性。

加上多态

当存在继承而且还存在 virtual function 的时候，事情就更加有趣了。数据应该怎么存放呢？唔。。特别是假如有人使用 base class 的指针指向 derived class 或者用 derived class 初始化 base class，模型需要保证不会出问题。

每一个类都有一个 vptr 指针。在深入探索C++对象模型书里，它的模型是 vptr 放在尾部，但是我是用 g++ 测试之后，vptr 是在头部的，而且不论继承多少次，一个 class object 里面只有一个放在头部的 vptr。个人感觉，放在头部的 vptr 更为合理，但是有一个问题，如果一个 base class 以一个 derived class 进行初始化操作，那么编译器需要对 base class 的 vptr 进行处理，以指向正确的 vtbl。

当去除 derived 的 virtual 时，也是一样的，所以可以看出继承链的 class 都只有一个 vptr。

虚拟继承

当谈到虚拟继承的时候，我甚至想不到内存模型应该是怎么样子的，因为要同时兼具低内存占用和高访问速度，这个好像挺难实现的，暂且看看书中是怎么描述的吧。

如果是一个 derived class 虚拟继承了一个 base class，那么这个 derived class 里面会有一个指针指向 base class。

一个很好的想法，可是应该是有一些问题，例如：

  • 每一个对象必须针对其每一个 virtual base class 背负一个额外的指针。
  • 由于虚拟继承串链的加长，导致间接存取层次的增加。例如我们有三层虚拟派生，我们就需要三次间接存取。

事实是这样吗，且让我们做个测试：

在测试的时候我并没有发现存在指向 virtual base class 的指针。。。

那应该就是第二种模型了，即把 virtual base class offsets 放入 vptr 指向的 vtbl 里。可是有两个虚基类怎么办？暂时还没有想到怎么测试 vtbl 的内容，也许能通过指针跟踪，但是最近项目太多了，就探索到这里吧 (^_^)。

这里有一个小技巧，如果是 virtual base class，那么它最好不带 data member，因为访问它的速度会比较慢。事实上，继承而来的 data member 是直接存放在 class object 里面的，虚拟继承所带来的主要冲击是，它会妨碍优化的有效性，因为每一层虚拟继承都会导入一个额外层次的间接性。

引入虚拟指针总会带来代价的，但是我在测试的时候，感觉应该没有这个代价了。。。
